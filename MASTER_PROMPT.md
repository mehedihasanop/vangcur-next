# Vangcur Next.js Migration — Master Prompt
(এই পুরো ফাইলের লেখা প্রতি নতুন Claude চ্যাটে কপি-পেস্ট করো)

---

## প্রজেক্ট পরিচিতি

আমি **Vangcur (ভাঙচুর)** নামে একটা বাংলাদেশি গ্যাজেট ও লাইফস্টাইল অ্যাক্সেসরিজ ই-কমার্স সাইট চালাই।
- বর্তমান সাইট: vanilla JavaScript + HTML/CSS, single large `index.html` (~12,000 লাইন) + আলাদা `admin.html`
- ব্যাকএন্ড: Supabase (products, orders, reviews, store_settings, customers ইত্যাদি টেবিল)
- হোস্টিং: Vercel/Netlify
- লক্ষ্য: পুরো প্রজেক্টটাকে **Next.js**-এ রূপান্তর করা — কারণ (১) দ্রুত পেজ লোড, (২) Supabase-এর উপর কম লোড/বেশি ভিজিটর হ্যান্ডেল করা, (৩) ভালো আর্কিটেকচার/সিকিউরিটি।

## কোডিং সম্পর্কে আমার সীমাবদ্ধতা — গুরুত্বপূর্ণ

- আমার ভালো ল্যাপটপ নেই, আমি নিজে কোড লিখতে/এডিট করতে পারি না।
- আমি শুধু: ফাইল কপি-পেস্ট করতে, GitHub-এ commit/push করতে, এবং Vercel-এ deploy দেখতে পারি।
- **সব কোডিং, ফাইল স্ট্রাকচার ডিজাইন, ডিবাগিং — সব তোমাকে (Claude) করতে হবে।**
- আমি নিজে Supabase ড্যাশবোর্ড থেকে Environment Variables ইত্যাদি সেট করতে পারব, যদি step-by-step বলে দাও।

## প্রজেক্ট স্থিতি ট্র্যাক করার সিস্টেম

- GitHub রিপো: **[এখানে রিপো লিংক বসাও]**
- রিপোর রুটে একটা `PROGRESS.md` ফাইল আছে যেখানে এখন পর্যন্ত কী হয়েছে, কী বাকি, এবং পরবর্তী ধাপ কী — তার বিস্তারিত লেখা থাকে।

### তোমার (Claude) প্রথম কাজ — প্রতি নতুন চ্যাটে:
1. উপরের GitHub রিপো লিংক থেকে repo ক্লোন/ফেচ করো (bash_tool দিয়ে `git clone`)।
2. `PROGRESS.md` পড়ো — এখান থেকে বুঝবে এখন পর্যন্ত কী হয়েছে, কোন আর্কিটেকচার ডিসিশন নেওয়া হয়েছে, এবং "Next Steps" সেকশনে কী করতে হবে।
3. বর্তমান কোড স্ট্রাকচার (ফোল্ডার/ফাইল) দেখো যাতে existing pattern অনুসরণ করো — নতুন করে নিজের স্টাইলে শুরু করো না।
4. কাজ শুরু করার আগে আমাকে ছোট করে বলো: "PROGRESS.md পড়েছি, এখন এই কাজটা করছি" — তারপর কাজ শুরু করো।

## কাজ করার নিয়ম (Architecture decisions — এগুলো বদলাবে না)

- **Routing**: Next.js App Router (`app/` ফোল্ডার ভিত্তিক)।
- **ভাষা**: JavaScript (TypeScript না, কারণ পরে আমি নিজে simple এডিট করতে চাইতে পারি, TS এর extra complexity দরকার নেই)।
- **স্টাইলিং**: বর্তমান CSS (`index.html`-এর `<style>` থেকে নেওয়া) যতটা সম্ভব আলাদা `.css`/`.module.css` ফাইলে নিয়ে reuse করো — Tailwind-এ পুরো রিরাইট করো না, এটা সময় ও রিস্ক বাড়াবে।
- **Supabase**:
  - পাবলিক ডেটা (প্রোডাক্ট লিস্ট, প্রোডাক্ট ডিটেইল, রিভিউ) → Server Components + ISR (revalidate: 60s)
  - ইউজার-নির্ভর/রিয়েল-টাইম (কার্ট, লগইন, অর্ডার প্লেস, অ্যাডমিন অ্যাকশন) → Client Components, supabase-js client-side (এখনকার মতোই)
  - Supabase URL ও anon key → `.env.local` / Vercel Environment Variables-এ রাখবে, কোডে hardcode করবে না।
- **অ্যাডমিন প্যানেল**: একইভাবে কাজ করবে (Supabase-এ direct read/write), শুধু React component-এ রূপান্তর হবে। কোনো নতুন ফাইল-ভিত্তিক ডেটা সিস্টেম না।

## ডেভেলপমেন্ট ও ডেলিভারি প্রসেস

1. কাজ sandbox-এ করবে, `npm run build` দিয়ে চেক করবে এরর নেই।
2. কাজ শেষে আমাকে বলবে কোন কোন ফাইল নতুন/পরিবর্তিত হয়েছে এবং সেগুলো present_files দিয়ে দেখাবে।
3. **`PROGRESS.md` আপডেট করবে** — আজ কী হলো, কী টেস্ট করতে হবে, পরের ধাপ কী।
4. আমি ফাইলগুলো রিপোতে push করব এবং Vercel preview লিংকে টেস্ট করে রিপোর্ট করব।
5. **লাইভ ডোমেইন (vangcur.com)-এ এখনও কিছু পরিবর্তন হবে না** — যতদিন না নতুন সাইট সম্পূর্ণ টেস্টেড।

## টোকেন/লিমিট সচেতনতা

- চ্যাট লিমিট হঠাৎ শেষ হয়ে যেতে পারে। তাই **বড় কাজ শুরু করার আগে ছোট ছোট sub-task-এ ভাগ করো**, এবং প্রতিটা sub-task শেষ হলেই PROGRESS.md আপডেট করে ফাইল present করো (পুরো বড় কাজ শেষে একসাথে দেবে না — মাঝে মাঝে checkpoint দাও)।
